# Схема обновления Hermes Agent (форк Maxim)

> Дата создания: 2026-05-23  
> Файл: `mx_hermes_changes/mx_hermes_update.md`

---

## 1. Обзор архитектуры

Hermes Agent установлен из **форка**, а не из официального репозитория напрямую. Это позволяет сохранять локальные изменения (коммиты) и при этом получать обновления из upstream.

### Репозитории и ремоуты

| Ремоут    | URL                                                | Назначение                  |
|-----------|----------------------------------------------------|-----------------------------|
| `origin`  | `https://github.com/kzmx23/hermes-agent.git`       | Форк Maxim (основной)       |
| `upstream`| `https://github.com/NousResearch/hermes-agent.git`  | Официальный upstream        |

- **Рабочая директория:** `/home/max/.hermes/hermes-agent`
- **Ветка:** `main`, отслеживает `origin/main`
- Локальные коммиты хранятся как обычные git-коммиты в форке
- Папка `mx_hermes_changes/` содержит документацию по всем локальным изменениям

### Схема потока кода

```
upstream/main (NousResearch)
       |
       | git fetch upstream
       v
  upstream/main (локальный ремоут)
       |
       | git merge upstream/main
       v
  origin/main (kzmx23/hermes-agent) ←--- локальные коммиты уже здесь
       |
       | git push origin main
       v
  GitHub (форк)
       |
       | hermes gateway restart
       v
  Рабочий экземпляр на сервере
```

## 2. Процедура обновления (пошагово)

### Шаг 1: Получить обновления из upstream

```bash
cd /home/max/.hermes/hermes-agent
git fetch upstream
```

Если ремоут `upstream` ещё не добавлен:

```bash
git remote add upstream https://github.com/NousResearch/hermes-agent.git
git fetch upstream
```

### Шаг 2: Посмотреть, что нового

```bash
git log HEAD..upstream/main --oneline --decorate
```

Оценить объём изменений. Если изменений много — стоит просмотреть детальнее:

```bash
git log HEAD..upstream/main --stat
git diff HEAD..upstream/main --stat
```

### Шаг 3: Убедиться, что рабочее дерево чистое

```bash
git status
```

Если есть незакоммиченные изменения — закоммитить или спрятать их:

```bash
git stash    # если нужно временно убрать
git stash pop  # вернуть после merge
```

**Важно:** перед merge все локальные изменения должны быть закоммичены.

### Шаг 4: Выполнить merge

```bash
git merge upstream/main
```

Используется именно **merge**, а не rebase. Причина: merge сохраняет оригинальные хеши локальных коммитов и не переписывает историю. Это безопаснее.

### Шаг 5: Разрешить конфликты (если есть)

См. раздел 3 ниже.

Если конфликтов нет — merge завершится автоматически. Перейти к шагу 6.

### Шаг 6: Проверить результат

```bash
# Посмотреть, что вмерджилось
git log --oneline -10

# Убедиться, что нет незакоммиченных файлов
git status
```

### Шаг 7: Запустить тесты

Проверить синтаксис Python-файлов, затронутых обновлением:

```bash
# Список изменённых Python-файлов
git diff --name-only HEAD~1 HEAD -- '*.py'

# Проверка синтаксиса каждого файла
python3 -m py_compile <файл>
```

Если есть pytest-тесты:

```bash
# Запустить тесты (если применимо)
pytest tests/ -x --tb=short
```

### Шаг 8: Отправить в форк

```bash
git push origin main
```

### Шаг 9: Перезапустить шлюз

```bash
hermes gateway restart
```

### Шаг 10: Проверить работоспособность

```bash
# Health check
curl -s http://localhost:<порт>/health

# Логи
hermes gateway logs --tail 50
# или
journalctl -u hermes-gateway --no-pager -n 50
```

Убедиться, что:
- Шлюз запустился без ошибок
- Нет traceback в логах
- Health check возвращает OK
- Агент отвечает на запросы

## 3. Разрешение конфликтов

### Правила

1. **Никогда не принимать слепо `--theirs` или `--ours`.** Каждый конфликт нужно просмотреть вручную.
2. Приоритет: сохранить локальные изменения, но вобрать исправления и новые фичи из upstream.
3. Если конфликт касается локальных кастомизаций — сохранять локальную версию, но проверить, не изменился ли API upstream.

### Процедура

```bash
# Посмотреть файлы с конфликтами
git diff --name-only --diff-filter=U

# Открыть каждый файл и разрешить конфликт вручную
vim <файл>
# или
nano <файл>
```

В файле будут маркеры:

```
<<<<<<< HEAD
// ваш локальный код
=======
// код из upstream
>>>>>>> upstream/main
```

После разрешения:

```bash
git add <файл>
git commit   # завершить merge commit
```

### Если конфликт слишком сложный

Можно отложить merge:

```bash
git merge --abort
```

Разобраться с конфликтами в отдельной ветке:

```bash
git checkout -b merge-upstream-$(date +%Y%m%d)
git merge upstream/main
# разрешить конфликты, протестировать
git checkout main
git merge merge-upstream-$(date +%Y%m%d)
```

## 4. Что НЕ делать

| Действие                                  | Почему нельзя                                                                  |
|-------------------------------------------|--------------------------------------------------------------------------------|
| `hermes update`                           | Команда стэшит локальные диффы и может их потерять. Форк полностью заменяет эту команду. |
| `git rebase upstream/main`                | Rebase переписывает историю локальных коммитов, ломает синхронизацию с origin. |
| `git push --force` без причины            | Может потерять коммиты. Только если есть веская причина и никто другой не работает с форком. |
| `git merge --strategy-option theirs`      | Слепое принятие upstream перезапишет локальные изменения.                      |
| Редактировать код Hermes напрямую в проде | Изменения должны быть закоммичены в форк, чтобы не потеряться при следующем обновлении. |

## 5. Процедура отката (rollback)

Если после обновления что-то сломалось:

### Быстрый откат

```bash
cd /home/max/.hermes/hermes-agent

# Найти последний рабочий коммит
git log --oneline -20
# или посмотреть reflog
git reflog

# Откатиться к нужному коммиту
git reset --hard <хеш-коммита>

# Отправить в форк
git push origin main --force   # force допустим при откате

# Перезапустить шлюз
hermes gateway restart

# Проверить
curl -s http://localhost:<порт>/health
hermes gateway logs --tail 50
```

### Откат только merge-коммита

Если нужно отменить именно последний merge (и merge ещё не был запушен или можно force push):

```bash
# Найти merge commit
git log --oneline -5

# Отменить merge
git reset --hard HEAD~1   # если merge был последним коммитом

# Или точечно:
git revert -m 1 <хеш-merge-коммита>
```

## 6. Отслеживание локальных изменений

Все локальные изменения документируются в папке:

```
/home/max/.hermes/hermes-agent/mx_hermes_changes/
```

### Файлы в этой папке

| Файл                         | Содержание                                       |
|------------------------------|--------------------------------------------------|
| `mx_hermes_update.md`        | Этот документ — схема обновлений                 |
| Другие `.md` файлы           | Документация по конкретным локальным изменениям  |

### Принцип

- Каждое локальное изменение — отдельный коммит с понятным сообщением
- Сообщение коммита должно начинаться с `mx:` для идентификации (опционально)
- Документация в `mx_hermes_changes/` описывает ЧТО изменено и ПОЧЕМУ

## 7. Кастомизация конфигурации

### Где хранить настройки

Конфигурация Hermes Agent находится в:

```
~/.hermes/config.yaml
```

**Не нужно** менять код Hermes для изменения настроек. Всё, что можно настроить через конфиг, должно настраиваться через конфиг.

### Что менять в config.yaml

- Ремоуты, провайдеры, ключи API
- Настройки моделей (temperature, max_tokens и т.д.)
- Параметры шлюза (порт, хост)
- Включение/отключение фич
- Системные промпты

### Когда менять код

Только если нужно:
- Исправить баг в самом Hermes
- Добавить новую функциональность, которая невозможна через конфиг
- Адаптировать поведение под специфические требования

Любые изменения кода → коммит в форк → документация в `mx_hermes_changes/`.
